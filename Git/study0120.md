### **Git**
- **협업**, 개발자는 혼자 일하는 사람이 아니다
- **Remote Repository** **원격 저장소**
  - 원격 저장소 *서비스* - Git은 프로그램, 아래 서비스들은 그 프로젝트를 저장하는 곳, 혼동 주의
    - GitLab, GitHub, Bitbucket
      - GitHib는 개발자의 SNS 중 하나
      - **TIL** 형식으로 매일 내가 배운 것을 정리해서 올린다면 **문서화**하는 좋은 습관 만들 수 있다. (GitHub에 TIL 검색)
      - 네이버 D2 참고 자료(https://d2.naver.com/news/3435170)
  - remote
    - *오해 주의* GitHub 와 Local Repository는 연결된 것이 아니다. Push & Pull 진행 이외에는 서로의 존재 알 수 없음
    - 추가의 개념 O 연결의 개념 X
  - 원격 저장소(Repository)에는 commit이 올라가는 것
    - commit 이력이 없다면 push 할 수 없다.
  - git push / pull & clone
    - 회사에서 컴퓨터를 키고 가장 먼저 하는 것(전날 변경사항 전부 pull)
    - p.19 그림 **중요**
    - 드라이브와 Git의 개념 차이를 설명할 수 있어야 한다. (Git 은 버전만 관리)
    - p.27 그림 **중요**
    - push는 변경사항 만큼만 올린다. 또한 버전 관리 역사가 동일해야 한다.(동일하지 않으면 문제 발생 가능)
    - pull 또한 변경사항 만큼만 받는다.  
    - clone 받은 프로젝트는 이미 git init이 되어 있습니다.
    - GitHub에서 Repository를 만들 때 Add README설정의 ON과 OFF 설정은 상황에 따라 다름
      - On 설정시 Initial Commit이 생김
      - OFF 설정은 로컬에서 먼저 작업 중 원격 저장소를 추가할 때 선호
      - ON 설정은 Gitclone 을 먼저 진행한 후 로컬 작업을 시작하는 경우 선호 (SSAFY에선 보통 이 경우 선호)
    - git config => 커밋에 작성되는 서명일 뿐 (인증수단 X)
    - **git credential** 
      - => 원격 저장소 입장에서 본인에게 push하는 사용자가 원격저장소에 권한이 있는 사람인지 확인하는 절차
      - => 컴퓨터에 1개만 저장됨
      - => 컴퓨터의 "자격 증명" 메뉴에 저장됨
      - 윈도우 검색 (자격 증명 관리자) -> Windows 자격 증명
        - 컴퓨터에 자동으로 저장됨
        - SSAFY에서 자리 옮길때마다 지워줘야 한다.
  - gitignore
    - gitignore에 작성해둔 파일들은 원격 저장소에 잡히지 않는다. (gitignore파일 자체는 잡힌다.)
    - 처음 프로젝트를 만들 때 작성해야 하는 파일
    - 이미 올려 둔 파일을 뒤늦게 gitignore에 추가해도 cli에서 잡힌다
    - **gitignore.io** => gitignore 파일 내용을 채우는 자동화 사이트이다. 앞으로 자주 쓰기 때문에 즐겨찾기 설정하기 -> 윈도우, 파이썬, vscode 등 검색(https://www.toptal.com/developers/gitignore/)
- **Revert & Reset**
  - commit을 없었던 일로 되돌리는 과정
  - **Git Revert** : 변경사항을 안전하게 실행 취소할 수 있도록 도와주는 순방향 실행 취소 작업
  - **Git Reset** : 특정 commit으로 되돌아가는 작업
  - **Reset 주의사항**
    - **Remote에 올라간 commit은 절대 reset 해서는 안됨**
- **TIL 관리**
  - GitHub을 개인 포트폴리오로 활용하는 방법
    - TIL을 통해 내가 학습하는 것을 기록
      - 필기 + α (학습 과정, 해결 과정)
      - 나의 학습 과정이 들어가야 한다. (스스로 새로 배운 내용을 기록하는 공간이다. 단순하게 필기 백업 용도가 아니다. 그렇기에 사실 굉장히 어려운 과정 -> 나의 문제 해결 과정과 방식 GitHub를 통해서 면접관이 볼 수 있다.)
      - 선배 기수 참고
        - 나정현 (https://github.com/najung-h)
    - 개인, 팀 프로젝트 코드를 공유
    - 오픈 소스 프로젝트에 기여 
  - **'문서화'의 중요성**
    - 신입 개발자에게 가장 기대되는 역량
      - 자신이 경험한 사용법을 문서화해서 팀 내에 전파할 수 있는 능력을 반드시 길러야 한다.
